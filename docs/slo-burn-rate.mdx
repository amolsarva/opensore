---
title: "SLO Burn Rate Calculator"
description: "Compute error budget burn rate, remaining budget, and time-to-exhaustion from SLI data."
---

The `slo_burn_rate` tool calculates your error budget status from raw SLI measurements — no external API required. It tells you exactly how fast you're burning your budget and when you'll exhaust it.

## When to use it

- You receive an SLO breach alert and want to quantify severity
- You want to determine how long the incident must resolve before SLO is at risk
- You're comparing burn rates across multiple services to prioritise response

## Parameters

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| `slo_target` | ✓ | — | SLO target as a fraction (e.g. `0.999` for 99.9%) |
| `error_count` | ✓ | — | Number of failed requests in the observed period |
| `total_requests` | ✓ | — | Total requests in the observed period |
| `window_hours` | | `720` | SLO measurement window in hours (default 30 days) |
| `observed_hours` | | `window_hours` | Duration of your observation window |

## Understanding burn rate

A burn rate of `1.0` means you're consuming error budget at exactly the allowed pace. A burn rate of `14.4` means your 30-day error budget will be exhausted in **2 hours** — the threshold for a fast-burn alert.

```json
{
  "burn_rate": 14.4,
  "error_budget_pct_remaining": 42.0,
  "time_to_exhaustion_hours": 2.1,
  "compliance_pct": 99.8,
  "is_burning_fast": true
}
```

## Example

```bash
opensore investigate --alert-name "HighErrorRate" \
  --description "5xx error rate spiked to 2% for the last 30 minutes"
```

The investigation agent will call `slo_burn_rate` automatically when SLO impact is suspected.
